# ADR 0005 - Multi-schema DB + explicit-DDL Alembic baseline

Status: Accepted, 2026-05

## Context

SkillForge bundles several domains in one database: central business data, external identity
links, SkillBot operational state, and auth. Dumping everything into `public` mixes unrelated
concepts and blurs naming/access boundaries. At the same time Alembic should carry the migrations
without a purely autogenerated baseline producing fragile, hard-to-read DDL.

## Decision

**One DB, five Postgres schemas** by domain (see
[`DATABASE_SCHEMA.md`](../DATABASE_SCHEMA.md)):

- `core` - central business domain (party/person/company, students, tutors, subjects)
- `geo` - geographic reference data
- `ext` - links from external system ids to a `core.party`
- `bot` - SkillBot operational state
- `auth` - OAuth2 clients, secrets, scopes, audit

Migration convention:

- The **baseline (`0001_baseline`) is hand-written as explicit DDL** - readable and deliberate,
  not autogenerated. **Per-PR revisions are autogenerated** and curated.
- `migrations/env.py` creates all schemas **inside the migration transaction**
  (`CREATE SCHEMA IF NOT EXISTS`) before any tables are created.
- Gotchas the convention addresses: enums **shared across schemas** (e.g. `member_role` in `bot`)
  need careful definition, and **enum types must be dropped explicitly on downgrade** - Alembic
  doesn't clean them up automatically.

## Consequences

- Clear domain boundaries; models mirror them via `*Base` classes with `{"schema": ...}`.
- Migrations are traceable (explicit baseline) instead of an opaque autogen wall; downgrades are
  covered by migration tests (PR #25).
- Slightly more discipline per migration (enum drops, schema creation), in exchange for
  reproducible apply/downgrade roundtrips.

Related: [ADR 0002](0002-pooled-vs-migration-url.md) (why migrations use the direct DB URL).
